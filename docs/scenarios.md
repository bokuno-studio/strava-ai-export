# QA シナリオ

`docs/onboarding.md` の MVP スコープから派生した QA シナリオ集。
実装が進むごとに更新する。

入力方式が API ベースに変わったため、ZIP アップロード系シナリオは廃止。
UI は英語のみ（日本語版は作らない方針）。

---

## シナリオ: トップページ表示（未認可）

- 前提: 初回訪問・未ログイン
- 操作: アプリ URL（例: `http://localhost:3000`）にアクセス
- 期待結果:
  - 200 OK で HTML が返る
  - 「Strava データを AI 用 CSV に変換」等の主旨が一目で分かる
  - 「Strava で連携」ボタン（OAuth 起動）が見える
  - スマホ幅（375px）で崩れない

---

## シナリオ: OAuth 認可（ハッピーパス）

- 前提: Strava アカウントを持ち、ログイン済み
- 操作:
  1. トップで「Strava で連携」をタップ
  2. Strava の認可画面で許可
  3. アプリに戻る
- 期待結果:
  - Strava の認可ページにリダイレクトされる
  - 戻り先で「同期する」UI が出る
  - サーバ DB に暗号化 token と athlete_id が保存される
  - クライアント側にも認可済みフラグが残る（Cookie or localStorage）

---

## シナリオ: OAuth 拒否

- 前提: OAuth 認可画面まで遷移済み
- 操作: 「拒否」を選ぶ
- 期待結果:
  - アプリに戻って「連携がキャンセルされました」表示
  - 再試行ボタンが見える
  - サーバ DB に中途半端なレコードが残らない

---

## シナリオ: ホーム画面追加（任意・できる人向け）

- 前提: `manifest.json` 設置済み・本番デプロイ済み
- 操作:
  - iOS Safari: 共有 → 「Add to Home Screen」
  - Android Chrome: メニュー → 「Add to Home screen」
- 期待結果:
  - アイコン・アプリ名が `manifest.json` の設定通り（英語）
  - アイコンタップで `display: standalone` で起動（アドレスバーなし）
  - OAuth 状態が保持されている
- 注意: この操作は MVP の必須フローではない。**追加できない・しないユーザーも普通の Web として完全に使える**ことが本シナリオより優先

---

## シナリオ: 「最新 1 件」取得（運動直後・主要ハッピーパス）

- 前提: OAuth 済み・直近にアクティビティを記録した
- 操作: PWA を開いて「最新 1 件」をタップ
- 期待結果:
  - 進捗インジケータが出る
  - 約 1〜5 秒で取得完了（詳細 + ストリーム + ラップ + ゾーン = 4req）
  - 「ダウンロード」ボタンが現れ、1 件分の CSV ZIP が落ちる
  - IndexedDB にもデータが蓄積される

---

## シナリオ: 「直近 7 日」取得

- 前提: OAuth 済み・前回同期から数日経過
- 操作: 「直近 7 日」をタップ
- 期待結果:
  - 期間外の activity を再取得しない
  - 新規 activity を取得し、約 10〜20 秒で完了
  - 「ダウンロード」ボタンが現れ、差分 CSV ZIP が落ちる
  - 既存 activity に更新があれば上書き

---

## シナリオ: 「全期間」初回フル取得（ハッピーパス）

- 前提: OAuth 済み・データ取得未実施
- 操作: 「全期間」をタップ
- 期待結果:
  - 進捗インジケータが出る（取得済み件数 / 推定総件数）
  - 取得は Web Worker で動き UI フリーズしない
  - IndexedDB に全データが蓄積される
  - サーバ DB の `last_synced_at` と `sync_cursor` が更新される
  - 完了後「ダウンロード」ボタンが現れる

---

## シナリオ: 「全期間」取得中にタブを閉じる → 再開

- 前提: 全期間取得の途中（例 50% 完了）
- 操作: タブを閉じる → 同じブラウザで再訪 → 「全期間」を再タップ
- 期待結果:
  - 既取得分を重複取得しない（`sync_cursor` から再開）
  - 残り分のみ取得される
  - IndexedDB の既存データが消えない

---

## シナリオ: アプリ全体レート制限ヒット（混雑時）

- 前提: 他ユーザーの取得でアプリ全体の `100 req / 15min` が逼迫
- 操作: ユーザーがボタンタップ
- 期待結果:
  - `X-RateLimit-Usage` を見て 90% 超ならサーバが「混雑中、しばらく待ってください」表示
  - 制限解除後に自動リトライ or 「再試行」ボタン
  - クライアント側は永遠に待たされない（タイムアウト + ガイダンス）

---

## シナリオ: ユーザー単独レート制限ヒット

- 前提: 単一ユーザーが「全期間」取得中に `100 req / 15min` に到達
- 操作: そのまま放置
- 期待結果:
  - 自動 backoff
  - UI に「Strava の制限のため待機中…」表示
  - 制限解除後に自動再開
  - 1000 req/day も同様（翌日持ち越し）

---

## シナリオ: 差分 CSV ダウンロード

- 前提: 同期完了直後（初回 or 増分）
- 操作: 「ダウンロード」ボタンをタップ
- 期待結果:
  - `strava-ai-export-YYYYMMDD-HHmm.zip` 形式で保存される
  - 中身に少なくとも以下が含まれる:
    - `activities.csv`（差分の行）
    - `laps.csv`
    - `zones.csv`
    - `streams/<activity_id>.csv`（差分の件数だけ）
    - `gear.csv`
    - `athlete_profile.json`
    - `athlete_stats.json`
    - `athlete_zones.json`
    - `prompt_template.txt`
  - CSV の値が `=` `+` `-` `@` で始まる場合に `'` でエスケープされている
  - ネットワークタブで外部サーバへの大容量送信が発生していない（CSV 生成はクライアント）

## シナリオ: AI ごとのアップロード方法ガイド

- 前提: ダウンロード完了画面
- 操作: 画面を確認
- 期待結果:
  - ChatGPT / Claude / Gemini の各アップロード案内が表示される
  - ChatGPT は ZIP 直、Claude/Gemini は展開して CSV を投げる旨が区別されている

## シナリオ: 異常系 - token 失効

- 前提: 何らかの理由で refresh token も無効化されている
- 操作: 「同期する」をタップ
- 期待結果:
  - 「再認証が必要です」表示
  - 「Strava で再連携」ボタンで OAuth に戻れる
  - 既存の IndexedDB データは消えない

---

## シナリオ: 異常系 - Strava 障害

- 前提: Strava API が 5xx を返す状態
- 操作: 同期実行
- 期待結果:
  - 指数バックオフでリトライ（3〜5回）
  - それでもダメなら「Strava 側で問題が起きています」表示
  - アプリはクラッシュしない・後ほど再試行可能

---

## シナリオ: 異常系 - IndexedDB 容量超過

- 前提: ストリームデータが膨大でブラウザのストレージ上限に近づく
- 操作: 同期実行
- 期待結果:
  - 容量警告 UI（「残り XXMB です」）が出る
  - 同期を一時停止して「ダウンロードして領域を空けますか？」を提案
  - クラッシュしない

---

## シナリオ: モバイル UI（375px）

- 前提: iPhone SE 相当のビューポート
- 操作: OAuth → 同期 → ダウンロードまで通す
- 期待結果:
  - すべてのタップターゲットが 44px 以上
  - 横スクロールが発生しない
  - フォントが極小にならない

---

## シナリオ: セキュリティヘッダ

- 前提: 本番デプロイ済み環境
- 操作: `curl -I https://<URL>` でヘッダ確認
- 期待結果:
  - `Content-Security-Policy` / `X-Content-Type-Options` / `Referrer-Policy` が設定されている
  - Strava OAuth リダイレクト先のドメインが CSP の `form-action` 等で許可されている
  - `next.config.ts` の設定が反映されている

---

## シナリオ: token の保管

- 前提: 任意の連携済みアカウント
- 操作: サーバ DB の token カラムを直接確認
- 期待結果:
  - 平文で保存されていない（AES-256-GCM 等で暗号化）
  - 暗号化鍵は環境変数で管理されている
  - athlete_id 以外で他人の token を引けない（RLS or 等価制御）

---

## シナリオ: favicon / OG / sitemap / llms.txt

- 前提: 本番デプロイ済み環境
- 操作:
  - `/favicon.ico` を直接取得
  - OGP デバッガで OG 画像を確認
  - `/sitemap.xml` と `/llms.txt` を取得
- 期待結果: いずれも 200 OK で内容が妥当
